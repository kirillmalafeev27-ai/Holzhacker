# Forest Defense — Implementation Plan

## Existing project findings

The workspace contains a complete low-poly nature pack under `GLTF format/` and the finished chopping/falling source scene `tree_chop_simulation.blend`. There is no existing browser game, player controller, collision engine, GLB loader, German-question system, or survival-defense state machine in the repository.

The existing chopping work will be treated as locked source content. Its five tree states, axe timing, wood-chip/spark effect, and fall result will not be remodeled or rewritten in Blender. The game integration will only expose this result to pickup/carry/build logic.

## Systems to add

- Offline Canvas 2D isometric renderer with depth sorting and a fixed game camera.
- Blender-rendered transparent sprite atlas generated directly from the existing nature GLBs.
- Player movement, collision, interaction, carrying and dodge.
- Integration wrapper around the existing chopping stages and fall/log result.
- Four stable fort states driven by delivered logs.
- Base health, damage visuals, repair zone and repair feedback.
- 180-second attack timer plus a developer time-scale option.
- Five-goblin state machine (`spawn`, `approach`, `attack`, `hit`, `defeated`).
- Three asynchronous catapults with health, launch animation and ballistic projectiles.
- Player health, invulnerability, projectile warning marker and defeat.
- Tower interaction and stone stock.
- One unified German A1–A2 question system containing at least 20 shuffled questions.
- Ballistic tower stones, catapult damage/destruction and note drops.
- Six collectible notes, victory, player-death defeat and base-destruction defeat.
- HUD, interaction prompts, question dialog, notifications and restart flow.
- Deterministic test/debug controls for accelerated attack and reproducible screenshots/video.

## GLB creation decision

One new assembled GLB is required: `public/assets/forest_defense/forest_defense_location.glb`. It will contain the clearing ground, reused nature meshes, the construction-zone composition and named gameplay markers. Repeated dynamic objects remain separate in the game for state changes and performance.

No new tree, rock, grass, log, stump, fence, gate, tent, bed, campfire, pot or sign asset will be modeled because equivalents already exist.

No matching catapult, goblin, watchtower, note, projectile, player, barrel or crate GLB exists. These dynamic low-poly gameplay objects will be drawn procedurally by the isometric Canvas renderer rather than exported as unnecessary standalone GLBs.

The browser game remains fully offline. Existing GLBs are not replaced: Blender renders the tree, rock, grass, stump, log, fence, gate, tent and campfire sprites used by the game, and the same placed assets are preserved in the assembled location GLB.

## Verification strategy

- Build and reopen the assembled location GLB in Blender.
- Run JavaScript syntax checks and a production asset/path audit.
- Start the local server and exercise the game in the browser.
- Use developer time scaling only for validation; normal attack time remains 180 seconds.
- Capture all 16 required visual states and create `docs/forest_defense_contact_sheet.png`.
- Record a short complete gameplay demonstration to `docs/forest_defense_gameplay.mp4`.
- Complete scenarios A–I and document evidence in `FOREST_DEFENSE_REPORT.md`.
